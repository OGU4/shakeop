```bash
uv sync
```

### 実行

```bash
# メインアプリ
uv run python -m salmon_buddy

# ミニアプリ（例: exp_001 静止画テスト）
uv run python experiments/exp_001_baito_text_recognition/main.py \
    --image data/test_fixtures/text/positive/baito_001.png --debug
```

## 1. 開発の基本原則

### 1.1 ミニアプリ駆動開発 (Mini-App Driven Development)

```
【原則】
  機能は必ず「ミニアプリ」として単独で動作確認してから、メインアプリに統合する。
  いきなりメインアプリに機能を書かない。
```

**ミニアプリの定義:**

ミニアプリとは、1つの認識機能を `experiments/exp_NNN_xxx/` に実装した単独動作可能なアプリケーションである。以下の2段階で構成される:

1. **CLI版**: コマンドライン引数で静止画・カメラ入力を受け取り、認識結果をコンソールに出力する。テスト（精度・速度の検証）はこの段階で行う
2. **GUI組み込み**: CLI版で精度・速度の基準を満たした後、GUIビューワー（exp_003）にプラグインとして組み込んで実機と同等の表示・操作で動作確認する

テスト作成・実行はCLI版の段階で行う。GUI組み込み後はユーザーが手動テスト（ステップ7）で確認する。

**GUI組み込みの手順:** `experiments/exp_003_gui_recognition_viewer/recognition_plugin.py` に定義された `RecognitionPlugin` Protocol に準拠したプラグインファイルを作成し、`experiments/exp_003_gui_recognition_viewer/plugins/` ディレクトリに配置する。既存プラグイン（同ディレクトリ内の `.py` ファイル）を参考にすること。

※ 上記はいずれも `experiments/` 内で完結する。`src/salmon_buddy/` への **メインアプリ移植** は別の作業であり、セクション6.1の統合チェックリストに従う。

**なぜこうするか:**

- ゲーム画面認識は試行錯誤が多い。ミニアプリなら数秒で起動して即テストできる
- バグの原因が「認識ロジックの問題」か「統合の問題」かを切り分けられる
- 各ミニアプリが「この認識はこの精度で動く」という証拠（エビデンス）になる
- 後から別の手法に差し替えるときも、ミニアプリ単位で比較実験できる

### 1.2 守るべき4つのルール

| # | ルール | 理由 |
|---|---|---|
| **R1** | **実装の前に必ず管理番号を発行し、要求仕様書と機能設計書を書く** | 何を・なぜ・どう作るかを明文化してから手を動かす |
| **R2** | ミニアプリは**30秒以内に起動して動作確認できる**こと | 試行錯誤のサイクルを速くする |
| **R3** | ミニアプリとメインアプリは**共通のデータ型（Protocol/dataclass）を使う** | 統合時のインターフェース不一致を防ぐ |
| **R4** | ミニアプリのREADMEに**実験結果（精度・速度・課題）を必ず記録する** | 何を試して何がダメだったかの知識を残す |

## 2. 管理番号とドキュメント管理

### 2.1 管理番号の体系

すべての開発作業（機能追加・バグ修正・リファクタリング）に管理番号を付与する。

```
F-NNN   機能 (Feature)           例: F-001, F-002, F-010
B-NNN   バグ修正 (Bug)            例: B-001, B-002
R-NNN   リファクタリング (Refactor) 例: R-001, R-002
```

#### 管理番号の種類ごとの注意事項

**R-NNN（リファクタリング）の場合:**

リファクタリング案件でも開発フロー（ステップ1〜8）を適用する。要求仕様書と機能設計書は必須。

- **要求仕様書**: 「なぜリファクタリングするか」「変更前後で外部仕様（入出力・振る舞い）が変わらないこと」「受け入れ基準」を記述する
- **機能設計書**: 「どのモジュール・関数を変更するか」「変更前後の構造比較」「影響範囲」を記述する
- 何を書けばよいか不明な場合は、ユーザーにヒアリングしてから作成すること

**B-NNN（バグ修正）の場合:**

バグ修正案件でも開発フロー（ステップ1〜8）を適用する。要求仕様書と機能設計書は必須（なければ作成する）。

- **要求仕様書**: `docs/REQUIREMENTS_STANDARD.md` に従って作成する。通常の機能要件に加えて「問題の説明（再現手順）」「期待される動作」「重要度」を記述する
- **機能設計書**: `docs/DESIGN_STANDARD.md` に従って作成する。修正対象のモジュール・関数、原因分析、修正方針を記述する
- 何を書けばよいか不明な場合は、ユーザーにヒアリングしてから作成すること
- 仕様未定義の不具合は、不具合修正ではなく**機能追加（F-NNN）として扱う**（`docs/BUGFIX_STANDARD.md` 参照）

### 2.2 ドキュメント管理ディレクトリ

`docs/issues/` 配下に案件ごとのフォルダを作成する。ディレクトリ名は `{管理番号}_{短い説明}` とする。
各フォルダには `requirements.md`（要求仕様書）、`design.md`（機能設計書）、必要に応じて `notes/`（補足資料）を配置する。

### 2.3 開発作業の流れ

各機能について、以下のフローを**厳守**する。**planモード（`/plan` / EnterPlanMode）は使わない**（通常モードで調査・計画を行う）。Agent ツールの `subagent_type=Plan` は使用してよい。

> **注**: requirements.md・design.md 内部のステータスフィールドは**仕様書の完成度**を示すものであり、index.md のステータス（案件全体の進捗）とは独立して管理する。

1. **案件作成** → `docs/issues/{管理番号}_{slug}/` フォルダを作成し、`docs/issues/index.md` にステータス 📋 未着手 で追加する。`CLAUDE.md`「現在の作業状況」に当該タスクをステータス 📋 未着手 で追加する
2. **調査・計画** → `docs/issues/index.md` のステータスを 📝 仕様作成中 に更新する。通常モードで既存コードを調査し、要求仕様書（`docs/REQUIREMENTS_STANDARD.md` 準拠）と機能設計書（`docs/DESIGN_STANDARD.md` 準拠）を作成または更新する。requirements.md・design.md のステータスを `📝 作成中` に設定する。B-NNN（バグ修正）の初回はinvestigation.mdもこのステップで作成する（セクション2.7参照）。差し戻し（ステップ7→ステップ2）の場合は、ステータスを 📝 仕様作成中 に戻し、不備のあったドキュメントを修正する（全面書き直しを含む）
3. **ドキュメント保存** → 要求仕様書を `docs/issues/{案件フォルダ}/requirements.md`、機能設計書を `docs/issues/{案件フォルダ}/design.md` にファイル保存する。差し戻し時は `investigation.md` も保存する。**保存が完了するまで実装に進んではならない**
4. **レビュー（/review + 人）** → `/review` スラッシュコマンドでレビューを実行する。引数にはレビュー対象ファイルのパスをスペース区切りで渡す。
   - investigation.md がない場合: `/review docs/issues/{案件フォルダ}/requirements.md docs/issues/{案件フォルダ}/design.md`
   - investigation.md がある場合: `/review docs/issues/{案件フォルダ}/requirements.md docs/issues/{案件フォルダ}/design.md docs/issues/{案件フォルダ}/investigation.md`

   ユーザーも同時にレビューする。`/review` は `docs/REVIEW_CRITERIA.md` の基準に従い、保存済みの要求仕様書・機能設計書をレビューする。`investigation.md` が存在する場合はレビュー対象に含める（レビュー基準: `docs/REVIEW_CRITERIA.md` セクション10）
5. **修正（必要な場合）** → レビューで問題があれば、再調査してドキュメントを更新する。**ステップ2〜4を問題がなくなるまで繰り返す**
6. **実装** → レビュー通過後（ステップ6に進む直前）、requirements.md・design.md のステータスを `✅ 承認済` に更新する。レビュー承認済みの設計書に新規ライブラリが記載されている場合は、実装開始前に TECH_STACK.md を更新する（DESIGN_STANDARD.md §1.3 参照）。`docs/issues/index.md` のステータスを 🔬 実装中 に更新する。ドキュメント（要求仕様書・機能設計書・CLAUDE.md・TECH_STACK.md、および設計書が参照するドメイン知識文書）を読んで実装する。ミニアプリ（CLI版 → GUI組み込み）を作成し、ミニアプリ新規作成時はセクション4.2・4.4に従いREADME.mdを作成しステータスを「実験中」に設定する。テスト作成・実行はCLI版の段階で行う（セクション8のルールに従う）。メインアプリ移植が必要な場合はセクション6.1の統合チェックリストに従う。ステップ6の全作業（実装・テスト作成・test-runner全パス・コミット）が完了したら、Claude Codeはその旨をユーザーに報告し、ステップ7のテスト項目リスト提示に進む（ユーザーの指示を待たない）
7. **手動テスト（実機確認）** → `docs/issues/index.md` のステータスを 🧪 テスト中 に更新する。ユーザーが実機でテストする。テスト観点は以下の通り:
   - 要求仕様書の受け入れ基準をすべて満たしているか
   - 機能設計書に記載された正常系の動作が期待通りか
   - 機能設計書に記載された異常系・境界条件の動作が期待通りか
   - 実機環境（カメラ入力・照明条件等）で精度・速度が許容範囲内か

   Claude Codeはステップ7の開始時に、上記の観点に基づいた具体的なテスト項目リストを要求仕様書・機能設計書から抽出し、ユーザーに提示する。テスト項目リストはチャット内でチェックリスト形式（`- [ ]`）で提示する。ファイル保存はしない。構成は以下の通り:
   1. 受け入れ基準の各項目を1行ずつ転記する
   2. 機能設計書の正常系シナリオから、受け入れ基準に含まれていない動作確認項目を1行ずつ追加する
   3. 機能設計書の異常系・境界条件から、手動テストで確認可能な項目を1行ずつ追加する

   以下はすべて「不具合」として扱い、`docs/BUGFIX_STANDARD.md` に従う。**ユーザーの承認を得た上で、ステップ2〜7を繰り返す**（コード修正はステップ6で行う。ステップ7で直接コードを編集してはならない）。管理番号の種別（F-NNN / B-NNN / R-NNN）を問わず、差し戻し時は `docs/BUGFIX_STANDARD.md` に従い、ステップ2で `docs/issues/{案件フォルダ}/investigation.md` を作成する（`docs/BUGFIX_STANDARD.md` セクション3参照）
   - 不具合の発見
   - 要求通りに実装されていない
   - 要求仕様作成時のヒアリング漏れ

   ステップ7で発見された問題の分類:
   - **既存要求の不備（要求通りに実装されていない / ヒアリング漏れで要求が不足）**: 同一管理番号のまま差し戻し。requirements.md・design.md を修正してステップ2〜7を繰り返す
   - **既存要求と無関係な新規問題**: 新規管理番号を発行する。その問題に対応する要求仕様書の記述がどの案件にも存在しない場合は F-NNN（仕様未定義 = 機能追加）、別の既存案件の要求仕様書に該当する記述がある場合は B-NNN（仕様定義済みだが実装が不正 = バグ修正）とする。現在の案件は影響がなければステップ8（完了）に進む

   判断基準:
   - **同一管理番号（差し戻し）**: 当該案件の requirements.md に記載された機能要件・受け入れ基準の範囲内で発見された不備。要件の追加・修正で対応する
   - **新規管理番号**: 当該案件の requirements.md の機能要件・受け入れ基準の範囲外の問題。別の画面・別の認識対象・別の操作に関する問題
   - **迷う場合**: ユーザーに確認する

   BUGFIX_STANDARD.md の「仕様未定義の不具合は F-NNN として扱う」は、上記いずれのケース（独立バグ案件・ステップ7での新規発見）にも適用する。
8. **完了** → 以下をすべて実施する:
   - `docs/issues/index.md` のステータスを ✅ 完了 に更新する
   - ミニアプリの `README.md`「ステータス」をセクション4.4の定義に従い更新する:
     - **完了**: ミニアプリ(experiments/)での検証のみで完結した場合
     - **統合済み**: メインアプリ(src/salmon_buddy/)に移植した場合
   - ミニアプリの `README.md`「結果」セクションに最終的な精度・速度を記録する
   - ファイルの追加・削除があった場合は `CLAUDE.md` のディレクトリ構成を最新に更新する
   - `CLAUDE.md`「現在の作業状況」から当該タスクを削除する

### 2.4 管理番号一覧 (docs/issues/index.md)

フォーマットとステータス凡例は実ファイル `docs/issues/index.md` を参照。

### 2.5 要求仕様書 (requirements.md)

テンプレートと記述ルールは `docs/REQUIREMENTS_STANDARD.md` に従うこと。

### 2.6 機能設計書 (design.md)

文書構成と記述ルールは `docs/DESIGN_STANDARD.md` に従うこと。

### 2.7 バグ修正（B-NNN）の場合

B-NNNでも requirements.md と design.md は必須（なければ作成する）。
記述基準は `docs/REQUIREMENTS_STANDARD.md` と `docs/DESIGN_STANDARD.md` に従う。
何を書けばよいか不明な場合は、ユーザーにヒアリングしてから作成すること。

#### 3つのドキュメントの役割

| ドキュメント | 役割 | 作成タイミング |
|---|---|---|
| requirements.md | 正しい動作の定義（受け入れ基準を含む） | ステップ2（初回） |
| design.md | 修正の設計（対象モジュール・修正方針） | ステップ2（初回） |
| investigation.md | 原因分析・修正計画の履歴（`docs/BUGFIX_STANDARD.md` セクション1の項目に従う） | ステップ2（B-NNN初回、またはステップ7差し戻し時は種別不問。差し戻し時はイテレーション番号を付けて追記） |

- **独立バグ案件（B-NNN新規）**: requirements.md → design.md → investigation.md の順で作成する。investigation.md には `docs/BUGFIX_STANDARD.md` セクション1に従い、不具合の特定・原因分析・修正内容・影響範囲・確認方法を記録する
- **ステップ7差し戻し時**: investigation.md にイテレーション番号を付けて原因分析・修正計画を追記する（上書きしない）。requirements.md・design.md に不備があれば併せて修正する（`docs/BUGFIX_STANDARD.md` セクション3に従う）

B-NNNの requirements.md では、通常の機能要件に加えて以下を記述する:
- **問題の説明**: 何が起きているか、再現手順
- **期待される動作**: 本来どう動くべきか（要求仕様書・機能設計書の該当箇所を引用）
- **重要度**: 致命的 / 重要 / 軽微

### 2.8 管理番号とミニアプリの対応

管理番号とミニアプリ（実験）は 1:1 とは限らない。要求仕様書はF番号に1つ。ミニアプリは手法ごとに複数作ってよい。
機能設計書に「どのミニアプリで検証するか」を記載する。

### 2.9 案件の取り下げ

開発フローのどの段階でも、ユーザーの判断で案件を取り下げることができる。取り下げ時は以下をすべて実施する:

1. `docs/issues/index.md` のステータスを ❌ 取り下げ に更新する
2. 要求仕様書（`requirements.md`）の「備考」セクションに取り下げ理由を記載する。要求仕様書が未作成の場合は、`requirements.md` を新規作成し「基本情報」と「備考（取り下げ理由）」のみ記載する
3. `CLAUDE.md`「現在の作業状況」に当該タスクがあれば削除する
4. 実装済みコード（experiments/ 内のファイル等）は削除しない。ミニアプリの `README.md` がある場合は「ステータス」をセクション4.4の定義に従い「不採用」に更新する

すべてを1つのリポジトリで管理する（モノレポ）。ミニアプリとメインアプリは同じリポジトリ内に共存する。

- ディレクトリ構成: `CLAUDE.md` の「リポジトリ構成」を参照
- 依存関係・バージョン情報: `docs/TECH_STACK.md` と `pyproject.toml` を参照
- 共有コードの import: ミニアプリからもメインアプリからも `shared/` 内のモジュールを同じように import できる（セクション4.3 ルール4参照）

## 4. ミニアプリの作り方

### 4.1 ミニアプリの命名規則

`exp_{連番3桁}_{短い説明}/` とする。

### 4.2 ミニアプリの必須構成

- **必須**: `README.md`（目的・手法・結果・知見）、`main.py`（単独実行のエントリーポイント）
- **任意**: `requirements.txt`（追加依存がある場合のみ）、`train.py`（学習スクリプト）、`notes/`（スクショ・ログ・メモ）

### 4.3 main.py の必須ルール

すべてのミニアプリは以下のルールに従う。新規作成時は既存の `experiments/exp_NNN_xxx/main.py` を参考にすること。

1. **2つの入力モードを用意する**: `--camera <デバイス番号>`（リアルタイムテスト）と `--image <パス>`（固定画像での精度検証・デバッグ）の両方を argparse で定義する
2. **デバッグ表示の切り替え**: `--debug` フラグで OpenCV の可視化ウィンドウの表示/非表示を制御する
3. **キー操作**: `s` キーでスクリーンショットを `notes/` に保存（学習データ収集も兼ねる）、`q` キーで終了
4. **共有コードの import パス**: `sys.path.insert` で `PROJECT_ROOT / "shared"` と `PROJECT_ROOT / "src"` を追加する（`PROJECT_ROOT = Path(__file__).resolve().parents[2]`）
5. **処理の分離**: メイン認識処理は `process_frame(frame)` に、デバッグ描画は `draw_debug(frame, result)` に分離する
6. **共有型の使用**: `salmon_types` の Protocol/dataclass を import して使う
7. **docstring**: モジュール冒頭に目的・手法・実行コマンドを記載する

### 4.4 ミニアプリ README.md の必須セクション

README.md には以下のセクションを含めること。新規作成時は既存の `experiments/exp_NNN_xxx/README.md` を参考にする。

1. **ステータス**: 以下のいずれかを記載する
   - **実験中**: ミニアプリとして開発・検証中（ステップ6でREADME.md作成時に設定）
   - **完了**: ミニアプリ(experiments/)での検証のみで完結した場合（ステップ8で設定）
   - **統合済み**: メインアプリ(src/salmon_buddy/)に移植した場合（ステップ8で設定）
   - **不採用**: 案件が取り下げられた場合（セクション2.9で設定）

   上記ステータスに加え、統合先（統合済みの場合）、作成日、最終更新を記載する
2. **目的**: 何を検証するか（1〜3行）
3. **手法**: 使う技術・アルゴリズム・ライブラリ
4. **実行方法**: カメラ入力・静止画テストの実行コマンド
5. **結果**: 精度（対象・正解率・テスト画像数）と速度（処理時間・環境）
6. **課題・知見**: 試行錯誤で得た知見、うまくいかなかったこと、改善アイデア
7. **統合メモ**: メインアプリに統合する際の注意点

## 5. 共有インターフェース（契約）

ミニアプリとメインアプリの間で「入出力の形」を先に決めておく。
これが **契約 (Contract)** であり、統合をスムーズにする鍵。

- Protocol定義: `shared/salmon_types/protocols.py`
- データモデル: `shared/salmon_types/models.py`
- 定数: `shared/salmon_types/constants.py`

型チェッカー（mypy）が Protocol 適合をチェックするため、「統合したら動かない」を事前検出できる。

## 6. メインアプリへの統合手順

### 6.1 統合チェックリスト

ミニアプリをメインアプリに統合するときは、以下を**毎回確認する**:

```markdown
## 統合チェックリスト: F-NNN / exp_NNN → src/salmon_buddy/xxx/yyy.py

- [ ] 要求仕様書 (docs/issues/F-NNN/requirements.md) が書かれている
- [ ] 機能設計書 (docs/issues/F-NNN/design.md) が書かれている
- [ ] ミニアプリのREADMEに結果が記録されている
- [ ] 要求仕様書の受け入れ基準を満たしている（各案件の requirements.md を参照）
- [ ] Protocol に準拠している（mypy でチェック）
- [ ] ハードコードされたパス・閾値を config から読むように変更した
- [ ] デバッグ用コード（cv2.imshow, print等）を除去 or ログレベルで制御に変更
- [ ] ユニットテストを作成した（tests/unit/test_xxx.py）
- [ ] テスト用fixture画像を data/test_fixtures/ に追加した
- [ ] 統合テストで他モジュールとの結合を確認した
- [ ] ミニアプリのREADME「ステータス」を「統合済み」に更新
```

### 6.2 統合時の変換ルール

ミニアプリ → メインアプリへの移植時、以下の変換を行う:

1. ハードコードされたパス・閾値 → コンストラクタ引数（設定から注入）
2. `print()` → `logging`
3. `cv2.imshow()` → 削除（GUIが担当）
4. Protocol に型が一致することを確認（mypy でチェック）

## 8. デバッグとテストの戦略

### 8.1 3段階テスト

```
Level 1: ミニアプリ内テスト（手動）
  └── main.py --image / --camera で目視確認
  └── 最も頻繁に行う。素早く回す。
  └── ステップ6（CLI版）で実施する。

Level 2: ユニットテスト（自動）
  └── pytest tests/unit/
  └── 固定画像 (test_fixtures) に対する期待値テスト
  └── ステップ6（CLI版で精度・速度が基準を満たした後）で作成・実行する。

Level 3: 統合テスト（自動）
  └── pytest tests/integration/
  └── パイプライン全体の通しテスト
  └── セクション6.1の統合チェックリストに従い、メインアプリ統合時に作成・実行する。
```

### 8.2 テスト用固定画像 (test_fixtures)

ミニアプリで `s` キーを押して保存したスクリーンショットのうち、
テストに適したものを `data/test_fixtures/` にコピーして管理する。
構成は実ディレクトリ `data/test_fixtures/` を参照。

### 8.3 ユニットテストの書き方

既存テスト `tests/unit/` のコードを参考にする。

### 8.4 デバッグ支援ツール

`shared/` 内の既存ユーティリティを参照。

### 8.5 サブエージェントへの委譲ルール（Claude Code用）

Claude Code利用時、テスト・品質チェック・ドキュメントレビューはサブエージェントに委譲する。

```
★ テスト・品質チェックは必ずサブエージェント（test-runner）を使って実行すること。
★ メインエージェントが直接 pytest, ruff, mypy を実行してはならない。
★ コードを変更したら、必ずサブエージェントでテストを実行すること。
★ ドキュメントレビュー（ステップ4）はサブエージェント（Agentツール）で実行すること。
  ユーザーも並行してレビューする。
```

#### test-runner サブエージェントに委譲する作業
- `pytest`（ユニットテスト・統合テスト）
- `ruff check`（リンター）
- `ruff format --check`（フォーマットチェック）
- `mypy --strict`（型チェック）

#### accuracy-checker サブエージェントに委譲する作業
- ミニアプリの精度検証バッチ（test_fixturesに対するバッチテスト）

#### ドキュメントレビュー（/review スラッシュコマンド）
- ステップ4で、`/review` スラッシュコマンドを実行する。`/review` は内部的にサブエージェントとして動作し、`docs/REVIEW_CRITERIA.md` の基準に従ってレビューする
- ユーザーも同時にレビューする（`/review` のレビューとユーザーのレビューは並行して行う）

#### サブエージェントに委譲しない作業
- 要求仕様書・機能設計書の**作成**（メインエージェントが担当。人間の承認が必要）
- アーキテクチャの設計判断
- experiments/ への新ファイル作成（メインエージェントが担当）

#### 実装中のテスト実行サイクル
- コード変更のたびにtest-runnerを呼ぶ（修正→テスト→修正→テストのサイクル）
- テストが失敗したらメインエージェントが修正し、再度test-runnerで確認する
- すべてのテストがパスするまでこのサイクルを繰り返す

## 9. Git運用ルール

### 9.1 コミットメッセージ規約

```
[F-NNN] 機能の要求仕様書・機能設計書・実装
  例: [F-001] 要求仕様書を作成
  例: [F-001] 機能設計書を作成
  例: [F-001] シーン分類モデルの学習データ追加

[B-NNN] バグ修正
  例: [B-001] Linux カメラクラッシュの修正

[R-NNN] リファクタリング
  例: [R-001] pHash共通化 — cv2.img_hash→shared/recognition に統一

[exp:NNN] ミニアプリ関連の変更
  例: [exp:002] 信頼度閾値の調整 0.7→0.6

[integrate] 統合作業
  例: [integrate] F-001 exp_002 → recognition/scene_classifier.py

[app] メインアプリの変更（管理番号に紐づかない軽微な変更）
  例: [app] GUI: コンパクトモード追加

[shared] 共有コードの変更
  例: [shared] SceneType に SPECIAL_GIANT を追加

[test] テスト関連
  例: [test] scene_classifier のユニットテスト追加

[docs] ドキュメント
  例: [docs] DEVELOPMENT.md 管理番号ルールを追加

[ci] CI/CD
  例: [ci] GitHub Actions にlintジョブ追加
```

**ポイント**: 機能実装・バグ修正・リファクタリングのコミットは必ず管理番号（F-NNN / B-NNN / R-NNN）を先頭に付ける。

#### プレフィックスの優先順位

1つのコミットが複数のプレフィックスに該当する場合、以下の優先順位で1つを選ぶ:

1. **管理番号 `[F-NNN]` / `[B-NNN]` / `[R-NNN]`** — 管理番号に紐づく作業は常にこちらを使う
2. **`[exp:NNN]`** — 管理番号に紐づかないミニアプリ単独の変更（閾値微調整、デバッグ出力追加等）にのみ使う

例:
- F-004の実装としてexp_004にコードを書いた → `[F-004] Wave数判定のCLI版を実装`
- 管理番号に紐づかないexp_003のUI微調整 → `[exp:003] デバッグパネルのレイアウト調整`

### 9.2 コミットタイミング

以下のタイミングでコミットする:

- ステップ3完了時（ドキュメント保存後）
- ステップ5完了時（レビュー指摘によるドキュメント修正後。修正がなければスキップ）
- ステップ6の実装が完了し、test-runnerが全パスした時点
- ステップ8の完了作業後

ステップ7で差し戻された場合も、繰り返すステップ2〜7の中で上記の同じタイミングでコミットする。
