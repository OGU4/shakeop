# docs/REQUIREMENTS_STANDARD.md

## 要求仕様書 記述基準
本基準に従って要求仕様書を作成すること。

---

### 1. テンプレート

以下のテンプレートに従って要求仕様書を作成すること。

```markdown
# [{管理番号}] <機能名/修正名/リファクタリング名> — 要求仕様書

## 基本情報

| 項目 | 内容 |
|---|---|
| 管理番号 | {F-NNN / B-NNN / R-NNN} |
| 作成日 | YYYY-MM-DD |
| ステータス | 📋 未着手 / 📝 作成中 / ✅ 承認済 |
| 関連実験 | exp_NNN_xxx（なければ「-」） |

## 背景・目的

<なぜこの機能が必要か。どの課題を解決するか。>

## 不具合情報（B-NNN の場合のみ記述）

### 問題の説明
- 再現手順: <具体的な手順>
- 発生環境: <OS、カメラ設定等>

### 期待される動作
- <要求仕様書・機能設計書の該当箇所を引用>

### 重要度
- 致命的 / 重要 / 軽微

## 機能要件

### FR-1: <機能名>
- <具体的な振る舞いの記述>

### FR-2: <機能名>
- <具体的な振る舞いの記述>

（機能ごとにIDを振る。各機能に「何ができるか」「入力」「出力」を記述する）

## 非機能要件

### NFR-1: <項目名（例: 認識精度）>
- <具体的な数値基準>

### NFR-2: <項目名（例: 処理速度）>
- <具体的な数値基準>

（パフォーマンス、対応環境、信頼性など。具体的な数値で記述する）

## 制約・前提

- <使用必須のライブラリ・モジュール>
- <入力の前提条件（例: FHD 1920x1080）>
- <実行環境の制約>

## 受け入れ基準

- [ ] 基準1: <例: test_fixturesの全画像に対して精度100%>
- [ ] 基準2: <例: 処理時間1ms以内>
- [ ] 基準3: <例: ユニットテストが全て通る>

（この機能が「完了」と言えるための具体的条件。チェックボックス形式で記述する）

## 備考

<注意事項、参考実装、技術方針など>
```

#### R-NNN（リファクタリング）での適用ガイダンス

DEVELOPMENT.md セクション2.1の方針に従い、テンプレートを以下のように適用する。

| セクション | 適用方法 |
|---|---|
| 背景・目的 | なぜリファクタリングするか（技術的負債、保守性、パフォーマンス等） |
| 不具合情報 | 記述不要（セクションごと省略） |
| 機能要件 | 変更前の外部仕様（入出力・振る舞い）を引用し、変更後も同一であることを明記する |
| 非機能要件 | リファクタリングの目的を具体的な数値・条件で記述する（例: 関数行数を50行以下にする、循環依存を解消する） |
| 受け入れ基準 | 検証可能な条件のみ記述する（例: 既存テストが全てパスする、mypy --strictがエラー0件） |

---

### 2. 記述ルール

#### 2.1 禁止表現
以下の曖昧表現は使用禁止。必ず具体値・具体条件に置き換える:
- ❌「適切に処理する」→ ✅「3回リトライし、すべて失敗したらエラーダイアログを表示する」
- ❌「高速に動作する」→ ✅「30FPS以上で動作する」
- ❌「必要に応じて」→ ✅「ファイルサイズが100MBを超えた場合」
- ❌「〜など」「〜等」→ ✅ 列挙を完結させる、または「以下に限定する: A, B, C」
- ❌「ユーザーフレンドリーな」→ ✅ 具体的なUI要件として記述する

#### 2.2 一意性
- 1つの要求に2つ以上の意味を含めない
- 「AかつB」は要求を分割する
- 各要求は他の要求を読まなくても単独で意味が通ること

#### 2.3 検証可能性
- すべての要求に対して「どうテストすれば満たされたと確認できるか」が
  明確に想像できる記述にする
- 主観的な表現（使いやすい、見やすい等）は客観的指標に置き換える

#### 2.4 トレーサビリティ
- 各要求にユニークIDを付与する
- 機能設計書の各項目が、どの要求IDに対応するか追跡可能にする
